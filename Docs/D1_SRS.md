# **Software Requirements Specification (SRS)**

**ระบบร้องเรียนปัญหาและแจ้งเตือนข่าวสารภายในมหาวิทยาลัย**  
**(University Issue Reporting and Emergency Notification System)**

## **1.บทนำ (Introduction)** 

### **1.1 วัตถุประสงค์ (Purpose)**

เอกสารนี้ระบุความต้องการซอฟต์แวร์สำหรับ "ระบบร้องเรียนปัญหาและแจ้งเตือนข่าวสารภายในมหาวิทยาลัย" (University Issue Reporting and Emergency Notification System) ซึ่งเป็นระบบที่เชื่อมโยง นิสิต, บุคลากร, สโมสรคณะ, เจ้าหน้าที่มหาวิทยาลัย และผู้ดูแลระบบ เข้าด้วยกันเพื่อประสิทธิภาพในการแก้ไขปัญหาและสื่อสารข้อมูล

### **1.2  ขอบเขตของระบบ (Scope)**

ระบบร้องเรียนปัญหาและแจ้งเตือนข่าวสารภายในมหาวิทยาลัย ครอบคลุมฟังก์ชันการทำงานหลักที่สอดคล้องกับความต้องการของผู้ใช้ ดังนี้  
**1.ระบบบริหารจัดการอัตลักษณ์และการเข้าถึง (Identity and Access Management):**

*  การตรวจสอบสิทธิ์ผ่านระบบอีเมลหน่วยงาน และการควบคุมสิทธิ์ตามบทบาท (Role-based Access Control) สำหรับผู้ใช้งานทั้ง 5 กลุ่ม

**2.ระบบบันทึกและจำแนกรายละเอียดคำร้องเรียน (Issue Submission & Classification):**

* การระบุสถานที่เกิดเหตุผ่านระบบ รายการตัวเลือก (Dropdown List) เช่น ชื่ออาคาร หรือคณะ และรองรับการกรอกข้อมูลแบบอิสระ (Text Box) ในกรณีที่ไม่มีข้อมูลในรายการ (ตัวเลือก "อื่นๆ")  
* ระบบแนบหลักฐานมัลติมีเดีย (Image Attachment) ที่รองรับการถ่ายภาพและการเลือกรูปจากคลังภาพ

	**3.ระบบบริหารจัดการลำดับงาน (Workflow Management):**

* กระบวนการคัดกรองหมวดหมู่ และการกำหนดระดับความสำคัญเร่งด่วน (Urgency Level)  
* การส่งต่อและมอบหมายงานให้หน่วยงานเฉพาะทาง (Task Assignment)

	**4.ระบบแจ้งเตือนและติดตามผล (Notification & Monitoring):**

* การแจ้งเตือนสถานะแบบเรียลไทม์ และระบบ Dashboard สำหรับติดตามไทม์ไลน์ของคำร้อง  
* การสำรวจความพึงพอใจหลังเสร็จสิ้นกระบวนการ (Satisfaction Survey)

### **1.3 ภาพรวมเอกสาร (Document Overview)**

	เอกสารฉบับนี้จัดแบ่งเนื้อหาออกเป็นหัวข้อหลักเพื่อเป็นแนวทางในการพัฒนาและตรวจสอบความถูกต้อง ดังนี้

* **บทนำ (Introduction):** ข้อมูลพื้นฐาน วัตถุประสงค์ และขอบเขตโดยรวมของโครงการ  
* **คำอธิบายโดยรวม (Overall Description):** บริบททางการใช้งาน ลักษณะเฉพาะของผู้ใช้ทั้ง 5 บทบาท และข้อจำกัดของระบบ  
* **ความต้องการเฉพาะ (Specific Requirements):** รายละเอียดเชิงฟังก์ชัน (Functional Requirements) ที่วิเคราะห์จาก User Stories และเกณฑ์การยอมรับ (Acceptance Criteria)  
* **อินเทอร์เฟซภายนอก (External Interface Requirements):** การเชื่อมต่อกับระบบภายนอก เช่น ระบบส่งจดหมายอิเล็กทรอนิกส์  
* **ความต้องการด้านคุณภาพ (Non-functional Requirements):** มาตรฐานความปลอดภัย ความเร็วในการประมวลผล และการปฏิบัติตามกฎหมายคุ้มครองข้อมูลส่วนบุคคล (PDPA)

### **1.4 เอกสารอ้างอิง (References)**

* **IEEE Std 830-1998:** Recommended Practice for Software Requirements Specifications  
* **ISO/IEC 27001:** มาตรฐานระบบบริหารจัดการความมั่นคงปลอดภัยสารสนเทศ  
* **พระราชบัญญัติคุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562 (PDPA):** เกณฑ์การบริหารจัดการข้อมูลส่วนบุคคลภายในประเทศไทย  
* **Web Content Accessibility Guidelines (WCAG) 2.1:** มาตรฐานการออกแบบเว็บเพื่อให้ทุกคนเข้าถึงได้  
* **University Information Technology Policy:** นโยบายและข้อบังคับด้านเทคโนโลยีสารสนเทศของมหาวิทยาลัย

### **1.5 ข้อตกลงเอกสาร (Document Conventions)**

เอกสารฉบับนี้ใช้รูปแบบการเขียนและสัญลักษณ์ที่เป็นมาตรฐาน เพื่อให้ผู้พัฒนาและผู้ตรวจประเมินมีความเข้าใจที่ตรงกัน ดังนี้:

* **สัญลักษณ์ระบุความต้องการ (Requirement Identifiers):**  
  * **REQ-XXX:** หมายถึง ความต้องการเชิงหน้าที่ (Functional Requirements) ซึ่งระบุถึงสิ่งที่ระบบต้องทำได้ เช่น REQ-001 (การเข้าสู่ระบบ)  
  * **NFREQ-XXX:** หมายถึง ความต้องการที่ไม่ใช่เชิงหน้าที่ (Non-functional Requirements) ซึ่งระบุถึงคุณสมบัติทางคุณภาพ เช่น NFREQ-001 (ประสิทธิภาพ)  
* **ระดับความสำคัญ (Priority Levels):**  
  * **สูง (HIGH):** ฟังก์ชันหลักที่จำเป็นต่อการดำเนินงาน หากไม่มีระบบจะไม่สามารถทำหน้าที่หลักได้  
  * **ปานกลาง (MEDIUM):** ฟังก์ชันเสริมที่สำคัญ ซึ่งช่วยให้กระบวนการสมบูรณ์ยิ่งขึ้นแต่อาจพัฒนาหลังจากฟังก์ชันหลักได้  
  * **ต่ำ (LOW):** ฟังก์ชันเพิ่มเติมเพื่อความสะดวกสบายของผู้ใช้  
* **สถานะของความต้องการ (Requirement Status):**  
  * **อนุมัติแล้ว (Approved):** ข้อกำหนดนี้ผ่านการตรวจสอบและพร้อมสำหรับการนำไปพัฒนา  
  * **อยู่ระหว่างดำเนินการ (In-Progress):** ข้อกำหนดที่กำลังออกแบบรายละเอียดเชิงลึก  
  * **\[TBD\]:** (To Be Determined) ข้อมูลที่ยังระบุไม่ได้ในขณะนี้ หรือรอการยืนยันจากหน่วยงานที่เกี่ยวข้อง

## **2\.ภาพรวมของระบบ (Overall Description)**

### **2.1 มุมมองผลิตภัณฑ์ (Product Perspective)**

ระบบร้องเรียนปัญหาและแจ้งเตือนข่าวสารภายในมหาวิทยาลัย ถูกออกแบบมาเพื่อเป็น แพลตฟอร์มดิจิทัลส่วนกลาง ที่เชื่อมโยงการสื่อสารระหว่างผู้ใช้งาน (นิสิต/บุคลากร) และหน่วยงานบริหารจัดการ (สโม/เจ้าหน้าที่มหาวิทยาลัย ) โดยระบบนี้ไม่ใช่โปรแกรมอิสระแต่เป็นส่วนหนึ่งของระบบนิเวศดิจิทัลในมหาวิทยาลัย ซึ่งมีการเชื่อมโยงดังนี้:

* **การเชื่อมต่อกับระบบระบุตัวตน:** ใช้ข้อมูลจากการลงทะเบียนเพื่อตรวจสอบสิทธิ์สมาชิกของมหาวิทยาลัย  
* **การทำงานแบบ Real-time:** เชื่อมต่อกับระบบ Push Notification เพื่อส่งข่าวสารและแจ้งเตือนเหตุฉุกเฉินให้ถึงเป้าหมายภายใน 1 นาที  
* **ฐานข้อมูลส่วนกลาง:** จัดเก็บประวัติการร้องเรียน สถานะการดำเนินการ และคะแนนความพึงพอใจเพื่อใช้ในการวิเคราะห์ข้อมูลภาพรวม

### **2.2 ฟังก์ชันผลิตภัณฑ์ (Product Functions)**

* **ฟังก์ชันการจัดการบัญชีและสิทธิ์การเข้าถึง:** ประกอบด้วยการเข้าสู่ระบบ (Login), การออกจากระบบ (Logout), และการจัดการข้อมูลโปรไฟล์ส่วนตัว  
* **ฟังก์ชันการร้องเรียนและส่งข้อมูลปัญหา:** ประกอบด้วยการสร้างคำร้องเรียน, การระบุสถานที่เกิดเหตุ, การแนบรูปภาพประกอบ และการกำหนดระดับความเร่งด่วน (นิสิต, บุคลากร)  
* **ฟังก์ชันการจัดการและดำเนินงาน:** ประกอบด้วยการรับเรื่องร้องเรียน, การคัดกรองหมวดหมู่, การระบุผู้รับผิดชอบ (สโมสรคณะ, เจ้าหน้าที่มหาวิทยาลัย ) และการยกเลิกคำร้องเรียน เฉพาะสถานะรอดำเนินการ(นิสิต/บุคลากร)   
* **ฟังก์ชันการติดตามและประเมินผล:** ประกอบด้วยการติดตามสถานะการร้องเรียนผ่านหน้าจอ Dashboard, การตรวจสอบประวัติย้อนหลัง (History)(ทุกกลุ่มผู้ใช้งาน) และการประเมินความพึงพอใจหลังเสร็จสิ้นการแก้ไข (นิสิต)   
* **ฟังก์ชันการสื่อสารและการแจ้งเตือน:** ประกอบด้วยระบบการตั้งค่าช่องทางแจ้งเตือน และการรับแจ้งเตือนสถานะงานหรือเหตุฉุกเฉินแบบเรียลไทม์   
* **ฟังก์ชันการบริหารจัดการระบบและข้อมูลพื้นฐาน:** ประกอบด้วยการจัดการข้อมูลคณะ/สถานที่/หมวดหมู่คำร้อง, การสร้างรายงานสรุปผลประจำเดือน และการตรวจสอบสถิติภาพรวมระบบ (แอดมิน)

### **2.3 ประเภทผู้ใช้งาน (User Classes)** 

	**แบ่งกลุ่มผู้ใช้งานตามบทบาทหน้าที่และความรับผิดชอบในระบบออกเป็น 4 กลุ่มหลัก ดังนี้:**  
	**1.ผู้ร้องเรียน (นิสิต/บุคลากร) (กลุ่มผู้ใช้หลัก \- Primary):**

* **ลักษณะพื้นฐาน:** นิสิตและบุคลากรภายในมหาวิทยาลัยที่มีบัญชีอีเมลหน่วยงาน มีระดับความเชี่ยวชาญด้านเทคโนโลยีระดับทั่วไป  
* **ความถี่การใช้งาน:** ใช้งานเมื่อพบปัญหาหรือต้องการติดตามสถานะคำร้อง  
* **กิจกรรมหลัก:** แจ้งเหตุร้องเรียนพร้อมหลักฐานรูปภาพและเลือกสถานที่, ติดตามสถานะ, รับแจ้งเตือนข่าวสาร, และประเมินความพึงพอใจ (เฉพาะนิสิต)

	**2.สโมสรคณะ (Samo) (กลุ่มผู้ใช้เชิงปฏิบัติการ \- Operational):**

* **ลักษณะพื้นฐาน:** ตัวแทนนิสิตในแต่ละคณะ มีความเข้าใจในโครงสร้างและปัญหาภายในคณะของตนเองเป็นอย่างดี  
* **ความถี่การใช้งาน:** รายวัน เพื่อตรวจสอบคำร้องใหม่ที่เกิดขึ้นในคณะ  
* **กิจกรรมหลัก:** คัดกรองและตรวจสอบความถูกต้องของข้อมูล, กำหนดระดับความเร่งด่วน, แก้ปัญหาเบื้องต้นในระดับคณะอัปเดตสถานะการดำเนินงาน หรือส่งต่อเรื่องให้หน่วยงานส่วนกลาง

	**3.เจ้าหน้าที่มหาวิทยาลัย (University Officer) (กลุ่มผู้ใช้เชิงเทคนิค \- Staff):** 

* **ลักษณะพื้นฐาน:** เจ้าหน้าที่มหาวิทยาลัย  หรือหน่วยงานอาคารสถานที่ มีทักษะและความเชี่ยวชาญในการแก้ไขปัญหาเฉพาะทาง  
* **ความถี่การใช้งาน:** รายวัน ตามภาระงานที่ได้รับมอบหมาย  
* **กิจกรรมหลัก:** รับเรื่องต่อจากสโมสรคณะ, มอบหมายงานให้ทีมช่างหรือหน่วยงานย่อย, อัปเดตสถานะการดำเนินงาน, และสรุปผลการแก้ไขปัญหา

	**4.ผู้ดูแลระบบ (Admin) (กลุ่มผู้ใช้ระบบ \- Secondary):** 

* **ลักษณะพื้นฐาน:** ผู้ดูแลระบบหรือผู้รับผิดชอบระบบ มีความเชี่ยวชาญด้านเทคโนโลยีสูง  
* **ความถี่การใช้งาน:** รายวัน เพื่อดูแลความเรียบร้อยของระบบภาพรวม  
* **กิจกรรมหลัก:** จัดการข้อมูลพื้นฐาน (คณะ/อาคาร/หมวดหมู่), ตรวจสอบสิทธิ์ผู้ใช้งาน, ส่งประกาศแจ้งเตือนเหตุฉุกเฉิน, และตรวจสอบรายงานสถิติประจำเดือน

### **2.4 สภาพแวดล้อมการทำงาน (Operating Environment)**

	**a) ฮาร์ดแวร์เซิร์ฟเวอร์ (Server Hardware)**

* **Infrastructure:** ใช้งานผ่านระบบคลาวด์ (Cloud Computing) เพื่อรองรับการขยายตัวของข้อมูลคำร้อง  
* **หน่วยความจำ (RAM):** 8 GB RAM  
* **หน่วยประมวลผล (CPU):**   
* **พื้นที่เก็บข้อมูล:** 100 GB สำหรับจัดเก็บข้อมูลคำร้อง ข้อมูล Master Data ของสถานที่ และไฟล์รูปภาพหลักฐาน

	**b) ฮาร์ดแวร์ฝั่งผู้ใช้งาน (User Hardware)**

* **อุปกรณ์:** คอมพิวเตอร์ (PC/Laptop), แท็บเล็ต และสมาร์ทโฟน  
* **หน่วยความจำ (RAM):** ขั้นต่ำ 2 GB ขึ้นไป  
* **การเชื่อมต่อ:**  อินเทอร์เน็ตความเร็ว 2-5 Mbps ขึ้นไป  
* **อุปกรณ์เสริม:** กล้องถ่ายภาพ (สำหรับถ่ายภาพหลักฐานการร้องเรียน)

	**c) ระบบปฏิบัติการที่รองรับ (Operating Systems)**

* **ฝั่งเซิร์ฟเวอร์:** Ubuntu 22.04 LTS หรือ Amazon Linux  
* **ฝั่งผู้ใช้งาน (Desktop):** Windows 10/11 หรือ macOS 11 ขึ้นไป  
* **ฝั่งผู้ใช้งาน (Mobile):** iOS 15+ หรือ Android 11+


### **2.5 ข้อจำกัดในการออกแบบและการใช้งาน (Design and Implementation Constraints)**

	**a) ข้อจำกัดด้านเทคโนโลยี (Technology Constraints)**

* **ภาษาและเฟรมเวิร์ก:** พัฒนาส่วนหน้าบ้าน (Frontend) ด้วยเทคโนโลยีที่รองรับ Responsive Design (เช่น React หรือ Flutter) และส่วนหลังบ้าน (Backend) ด้วยภาษาที่รองรับการจัดการข้อมูลจำนวนมาก (เช่น Node.js หรือ Python)  
* **ฐานข้อมูล:** ใช้ระบบจัดการฐานข้อมูลเชิงสัมพันธ์ (Relational Database) เช่น MySQL หรือ PostgreSQL เพื่อจัดเก็บข้อมูลคำร้องและความสัมพันธ์ของสถานที่ (Dropdown) อย่างเป็นระบบ  
* **การรักษาความปลอดภัย:** ต้องใช้โปรโตคอล HTTPS ในการส่งผ่านข้อมูล และต้องมีการเข้ารหัสผ่าน (Hashing) ของผู้ใช้งานก่อนบันทึกลงฐานข้อมูล

	**b) ข้อจำกัดด้านเวลา (Schedule Constraints)**

* **ระยะเวลาการดำเนินงาน:**  
* **การทดสอบระบบ:** ต้องดำเนินการทดสอบระบบในสภาพแวดล้อมจริง (UAT) ร่วมกับกลุ่มตัวแทนนิสิตและเจ้าหน้าที่ก่อนการส่งมอบ

	**c) ข้อจำกัดด้านความมั่นคงปลอดภัยและกฎหมาย (Security and Legal Constraints)**

* **มาตรฐาน PDPA:** ระบบต้องมีมาตรการคุ้มครองข้อมูลส่วนบุคคล (PDPA) โดยเฉพาะชื่อผู้แจ้ง เบอร์โทรศัพท์ และรูปภาพหลักฐาน ห้ามเผยแพร่แก่ผู้ที่ไม่มีสิทธิ์เกี่ยวข้อง  
* **การเข้าถึงข้อมูล:** การเข้าถึงข้อมูลในฐานข้อมูลต้องแบ่งสิทธิ์ตามบทบาท (Role-based Access Control) ที่กำหนดไว้ในหัวข้อ 2.3 เท่านั้น

	**d) ข้อจำกัดด้านอินเทอร์เฟซและการออกแบบ (Interface Constraints)**

* **การเลือกสถานที่:** ระบบต้องจำกัดการระบุสถานที่ให้อยู่ในรูปแบบ รายการตัวเลือก (Dropdown List) ตามฐานข้อมูลอาคารของมหาวิทยาลัยเท่านั้น เพื่อป้องกันความผิดพลาดของข้อมูล (Data Integrity)  
* **ความง่ายในการใช้งาน:**  ต้องออกแบบตามมาตรฐาน UI/UX ที่เรียบง่าย เพื่อให้ผู้ใช้งานทุกกลุ่ม (โดยเฉพาะเจ้าหน้าที่และบุคลากร) สามารถใช้งานได้โดยไม่ต้องผ่านการอบรมที่ซับซ้อน

### **2.6 สมมติฐานและการพึ่งพา (Assumptions and Dependencies)**

	**a) สมมติฐาน (Assumptions)**

* **ความพร้อมของข้อมูลพื้นฐาน:** สมมติว่ามหาวิทยาลัยมีฐานข้อมูลรายชื่อคณะ อาคาร และห้องต่างๆ ที่เป็นปัจจุบัน พร้อมให้นำมาใส่ในระบบ Dropdown List ได้ทันที  
* **สิทธิ์การเข้าถึงอีเมล:** สมมตินิสิตและบุคลากรทุกคนมีบัญชีอีเมลของมหาวิทยาลัย (@buu.ac.th) ที่ใช้งานได้จริง เพื่อใช้ในการยืนยันตัวตน  
* **การเชื่อมต่อเครือข่าย:** สมมติว่าผู้ใช้งานสามารถเชื่อมต่ออินเทอร์เน็ตผ่านเครือข่าย Wi-Fi ของมหาวิทยาลัยหรือเครือข่ายมือถือได้ในขณะที่ต้องการแจ้งเหตุ  
* **การตอบรับของหน่วยงาน:** สมมติว่าเจ้าหน้าที่และสโมสรคณะมีความพร้อมในการเข้าใช้งานระบบเพื่อตรวจสอบและอัปเดตสถานะคำร้องตาม Workflow ที่วางไว้

	**b) การพึ่งพา (Dependencies)**

* **ระบบพิสูจน์ตัวตนกลาง (Central Authentication):** ระบบต้องพึ่งพา API หรือฐานข้อมูลบัญชีผู้ใช้ของมหาวิทยาลัยเพื่อตรวจสอบว่าผู้ที่ Login เป็นนิสิตหรือบุคลากรจริง (Critical Dependency)  
* **ระบบส่งอีเมล (Email Service Provider):** พึ่งพาบริการส่งจดหมายอิเล็กทรอนิกส์ (เช่น Gmail API หรือ SMTP Service) สำหรับการส่งรหัสผ่านใหม่ หรือการแจ้งเตือนสถานะผ่านอีเมล  
* **โครงสร้างพื้นฐานคลาวด์ (Cloud Infrastructure):** พึ่งพาผู้ให้บริการเซิร์ฟเวอร์ (เช่น AWS, Google Cloud หรือ Azure) ในการติดตั้งระบบและจัดเก็บฐานข้อมูล

## **4\. ความต้องการอินเทอร์เฟซภายนอก (External interface requirements)**

**a) ระบบยืนยันตัวตนมหาวิทยาลัย (University Authentication)** (แทนที่ Payment Gateway ในตัวอย่าง เพราะระบบเราเน้นเรื่อง Login)

* **University LDAP / Active Directory API**  
* **วิธีการ:** RESTful API  
* **การยืนยันตัวตน:** OAuth 2.0 หรือ API Key  
* **ข้อมูลที่แลกเปลี่ยน:** ตรวจสอบ Username/Password, ดึงข้อมูลสังกัด (คณะ/หน่วยงาน)

* **SMTP Service (ของมหาวิทยาลัย) หรือ SendGrid API**  
* **วิธีการ: SMTP / RESTful API**  
* **ฟีเจอร์หลัก:**  
  * แจ้งเตือนเมื่อรับเรื่อง (Submission Confirmation)  
  * แจ้งเตือนเมื่อสถานะเปลี่ยน (Status Update)  
  * แจ้งเตือนงานด่วน (Urgent Task Alert)

	**d) ระบบฐานข้อมูลสถานที่ (Master Location Data)** (ถ้ามี API แยกจากกองอาคารสถานที่)

* **University Facility API** \[Optional\]  
* **วิธีการ:** JSON Data Import  
* **ต้องการข้อมูล:** รายชื่ออาคาร, รายชื่อห้อง, รหัสสถานที่

### **4.4 อินเทอร์เฟซการสื่อสาร (Communication Interfaces)**

* **โปรโตคอล:** HTTPS (TLS 1.2 หรือ 1.3)  
* **รูปแบบข้อมูล:** JSON (JavaScript Object Notation)  
* **Base URL:** https://issue.university.ac.th/api/v1  
* **Rate Limit:** 100 คำขอ/นาที ต่อ User (ป้องกัน Spam)  
* **Pagination:** 20 รายการคำร้องต่อหน้า (สำหรับหน้า Dashboard)  
* **API Response Time:** \< 500 ms (สำหรับการโหลดข้อมูลทั่วไป)

**5.1.1 คำอธิบายและลำดับความสำคัญ (Description and Priority)** ระบบต้องรองรับการยืนยันตัวตนผ่านอีเมลมหาวิทยาลัย (@buu.ac.th) เพื่อคัดกรองบทบาทผู้ใช้ (นิสิต/บุคลากร/สโมสร/เจ้าหน้าที่) และอนุญาตให้ผู้ดูแลระบบจัดการสิทธิ์หรือปิดกั้นผู้ใช้งานได้

* **ลำดับความสำคัญ:** สูง (HIGH)

**5.1.2 สถานะและขั้นตอน (Stimulus/Response Sequences)**

* **เข้าสู่ระบบ (Login):** ผู้ใช้กรอกอีเมล/รหัสผ่าน ➜ ระบบตรวจสอบกับฐานข้อมูลมหาวิทยาลัย ➜ หากถูกต้อง ระบบสร้าง Session และตรวจสอบ Role ➜ เปลี่ยนเส้นทางไปยัง Dashboard ตามบทบาท (เช่น นิสิตไปหน้าแจ้งเรื่อง, สโมสรไปหน้าคัดกรอง)  
* **แก้ไขโปรไฟล์ (Edit Profile):** ผู้ใช้กดแก้ไขข้อมูล ➜ เปลี่ยนรูปภาพ/เบอร์โทร ➜ กดบันทึก ➜ ระบบอัปเดตฐานข้อมูล ➜ แสดงข้อความ "บันทึกสำเร็จ"  
* **จัดการสิทธิ์ (Admin):** Admin ค้นหาชื่อผู้ใช้ ➜ กดเปลี่ยน Role (เช่น จากนิสิตเป็นสโมสร) ➜ ระบบบันทึก Log ➜ สิทธิ์การเข้าถึงของผู้ใช้เปลี่ยนทันที

**5.1.3 ความต้องการที่เกี่ยวข้อง (Related Requirements)**

* REQ-001: การเข้าสู่ระบบด้วยอีเมลมหาวิทยาลัย (University Auth Login)  
* NFREQ-002: ความปลอดภัยและกฎหมาย (Security & PDPA)

### **5.2 ฟีเจอร์ที่ 2: การแจ้งเหตุและติดตามสถานะ (Issue Reporting & Tracking)**

**5.2.1 คำอธิบายและลำดับความสำคัญ (Description and Priority)** ระบบต้องช่วยให้อำนวยความสะดวกในการแจ้งปัญหา โดยมีระบบช่วยระบุพิกัด (Dropdown/Text) และการแนบรูปภาพ รวมถึงการติดตามสถานะงานแบบ Real-time และการยกเลิกคำร้อง

* **ลำดับความสำคัญ:** สูง (HIGH)

**5.2.2 สถานะและขั้นตอน (Stimulus/Response Sequences)**

* **สร้างคำร้อง (Create Ticket):**  
  1. ผู้ใช้เลือกหมวดหมู่และสถานที่ (Dropdown) ➜  
  2. ผู้ใช้แนบรูปภาพ (กล้อง/คลังภาพ) ➜ ระบบแสดงรูปตัวอย่าง ➜  
  3. ผู้ใช้กดส่งเรื่อง ➜ ระบบสร้าง Ticket ID ➜ แจ้งเตือน "ส่งเรื่องสำเร็จ"  
* **ติดตามและยกเลิก (Track & Cancel):** ผู้ใช้ดู Timeline สถานะ ➜ หากสถานะเป็น "Pending" ปุ่มยกเลิกจะทำงาน ➜ ผู้ใช้กดยกเลิก ➜ ระบบเปลี่ยนสถานะเป็น Cancelled ➜ จบกระบวนการ  
* **ประเมินผล (Survey):** สถานะงานเปลี่ยนเป็น "Completed" ➜ ระบบแสดงปุ่ม "ประเมินความพึงพอใจ" (เฉพาะนิสิต) ➜ ผู้ใช้ให้ดาว ➜ บันทึกคะแนนลงระบบ

**5.2.3 ความต้องการที่เกี่ยวข้อง (Related Requirements)**

* **REQ-002: การสร้างคำร้องเรียนและการระบุสถานที่**  
* **REQ-003: การยกเลิกคำร้องเรียน**  
* **REQ-006: การประเมินความพึงพอใจ**

**5.3 ฟีเจอร์ที่ 3: กระบวนการจัดการและแก้ไขปัญหา (Workflow & Incident Management)**

**5.3.1 คำอธิบายและลำดับความสำคัญ (Description and Priority) ระบบต้องรองรับ Workflow การทำงานหลายขั้นตอน ตั้งแต่การคัดกรองโดยสโมสรคณะ การกำหนดความเร่งด่วน และการส่งต่อ/จบงานโดยเจ้าหน้าที่มหาวิทยาลัย**

* **ลำดับความสำคัญ: สูง (HIGH)**

**5.3.2 สถานะและขั้นตอน (Stimulus/Response Sequences)**

* **การคัดกรอง (Screening \- Samo):** มีคำร้องใหม่เข้ามา ➜ สโมสรคณะตรวจสอบข้อมูล ➜ (ทางเลือก A) แก้ไขหมวดหมู่/ระดับความเร่งด่วน ➜ (ทางเลือก B) กด "ส่งต่อ" ไปยังส่วนกลาง ➜ สถานะเปลี่ยนเป็น In Progress  
* **การแก้ไขงาน (Resolution \- Officer):** เจ้าหน้าที่ได้รับแจ้งเตือน ➜ กด "รับเรื่อง" (Assign to self) ➜ ดำเนินการซ่อมแซม ➜ กด "จบงาน" (Complete) และแนบรูปหลังซ่อม ➜ ระบบส่งเมลแจ้งผู้แจ้ง  
* **การจัดการ Master Data (Admin):** Admin เพิ่มชื่ออาคารใหม่ ➜ ระบบอัปเดต Dropdown ในหน้าแจ้งเรื่องทันที

**5.3.3 ความต้องการที่เกี่ยวข้อง (Related Requirements)**

* REQ-004: การคัดกรองและส่งต่องาน  
* REQ-007: การออกรายงานภาพรวม

**5.4 ฟีเจอร์ที่ 4: ระบบการสื่อสารและแจ้งเตือน (Notification & Communication)**

**5.4.1 คำอธิบายและลำดับความสำคัญ (Description and Priority)** ระบบต้องทำการส่งข้อมูลข่าวสารและสถานะงานไปยังผู้ใช้ผ่านช่องทางต่างๆ (Email, Dashboard) โดยอัตโนมัติเมื่อเกิดเหตุการณ์ (Trigger) ที่กำหนด

* **ลำดับความสำคัญ:** ปานกลาง (MEDIUM)

**5.4.2 สถานะและขั้นตอน (Stimulus/Response Sequences)**

* **แจ้งเตือนสถานะเปลี่ยน (Status Update Alert):** เจ้าหน้าที่กดอัปเดตงาน ➜ ระบบตรวจสอบ ID ผู้แจ้ง ➜ ระบบส่ง Request ไปยัง Email Server ➜ ผู้แจ้งได้รับอีเมล  
* **แจ้งเตือนเร่งด่วน (Urgent Alert):** บุคลากรระบุระดับ "ด่วนที่สุด" ➜ ระบบส่ง Push Notification/Email หาหัวหน้างานทันที  
* **ประกาศจากระบบ (System Broadcast):** Admin พิมพ์ประกาศปิดปรับปรุง ➜ กดส่ง ➜ ผู้ใช้ทุกคนได้รับ Notification บนหน้า Dashboard

**5.4.3 ความต้องการที่เกี่ยวข้อง (Related Requirements)**

* REQ-005: การแจ้งเตือนและติดตามสถานะ  
* NFREQ-001: ประสิทธิภาพ (การส่งเมลภายใน 1 นาที)

## **6\. ความต้องการอื่นๆ ที่ไม่ใช่เชิงหน้าที่ (OTHER NONFUNCTIONAL REQUIREMENTS)**

### **6.1 กฎทางธุรกิจ (Business Rules)**

	**BR-001: ลำดับขั้นของสถานะคำร้อง (Workflow State Transition)**

* สถานะเริ่มต้นต้องเป็น "รอดำเนินการ (Pending)" เสมอ  
* คำร้องจะเปลี่ยนเป็น "เสร็จสิ้น (Completed)" ได้ก็ต่อเมื่อ "เจ้าหน้าที่มหาวิทยาลัย" หรือ "สโมสรคณะ" เป็นผู้กดจบงานเท่านั้น (นิสิตผู้แจ้งไม่มีสิทธิ์กดจบงานเอง)  
* หากมีการส่งต่อ (Forward) จากสโมสรไปยังส่วนกลาง สิทธิ์การแก้ไขสถานะจะถูกโอนไปยังส่วนกลางทันที

	**BR-002: นโยบายการยกเลิกและแก้ไข (Cancellation & Editing Policy)**

* **การยกเลิก:** ผู้แจ้งสามารถกดยกเลิกคำร้องได้เฉพาะเมื่อสถานะยังเป็น "รอดำเนินการ" เท่านั้น หากเจ้าหน้าที่รับเรื่องแล้ว ("กำลังดำเนินการ") จะไม่สามารถยกเลิกได้  
* **การแก้ไขข้อมูลส่วนตัว:** ข้อมูลหลักที่ยืนยันตัวตน (รหัสนิสิต, ชื่อ-นามสกุล, คณะสังกัด) จะถูกล็อก (Read-only) ตามฐานข้อมูลมหาวิทยาลัย ไม่อนุญาตให้แก้ไขเองเพื่อป้องกันการปลอมแปลง

	**BR-003: สิทธิ์การเข้าถึงข้อมูลข้ามหน่วยงาน (Cross-Department Access)**

* "สโมสรคณะ" จะเห็นข้อมูลคำร้องเฉพาะของคณะตนเองเท่านั้น ไม่สามารถดูข้อมูลของคณะอื่นได้  
* "เจ้าหน้าที่มหาวิทยาลัย (ส่วนกลาง)" สามารถดูและจัดการคำร้องได้ทุกคณะ หากคำร้องนั้นถูกส่งต่อมาหรืออยู่ในความรับผิดชอบของตน

### **6.2 ความต้องการด้านข้อมูล (Data Requirements)**

	**DR-001: การบันทึกข้อมูลและหลักฐาน (Data Capturing)**

* **รูปภาพ:** ระบบต้องจัดเก็บรูปภาพหลักฐาน (Before) และรูปภาพหลังแก้ไข (After) โดยเชื่อมโยงกับ Ticket ID เสมอ  
* **พิกัดสถานที่:** ต้องบันทึกข้อมูลอาคารและห้องในรูปแบบ Structured Data (ID) เพื่อให้สามารถสรุปสถิติแยกตามอาคารได้แม่นยำ (ไม่ใช่แค่ Text)

	**DR-002: การบันทึกประวัติการใช้งาน (Audit Trail & Logging)**

* ระบบต้องบันทึก Log ทุกครั้งที่มีการเปลี่ยนแปลงสิทธิ์ผู้ใช้งานโดย Admin (Who, When, What)  
* ระบบต้องบันทึกประวัติการเปลี่ยนตัวผู้รับผิดชอบงาน (Re-assignment Log) เพื่อตรวจสอบความรับผิดชอบย้อนหลังได้

	**DR-003: ระยะเวลาการเก็บข้อมูล (Data Retention)**

* ข้อมูลคำร้องและรูปภาพจะถูกเก็บไว้อย่างน้อย 4 ปีการศึกษา (เพื่อให้ครอบคลุมระยะเวลาศึกษาของนิสิต 1 รุ่น)  
* หลังจากครบกำหนด ข้อมูลจะถูก Archive หรือลบรูปภาพออกเพื่อประหยัดพื้นที่ (เก็บเฉพาะ Text)

### **6.3 กฎหมาย/การปฏิบัติตามกฎระเบียบ (Legal/Compliance)**

	**LC-001: การคุ้มครองข้อมูลส่วนบุคคล (PDPA Compliance)**

* ชื่อ เบอร์โทรศัพท์ และอีเมลของผู้แจ้งเรื่อง จะต้องถูกเปิดเผยให้เห็นเฉพาะ "เจ้าหน้าที่ผู้รับผิดชอบงาน" เท่านั้น บุคคลภายนอกหรือนิสิตคนอื่นห้ามเห็น  
* ต้องมีกล่อง Consent ยอมรับเงื่อนไขการใช้งานก่อนส่งคำร้องครั้งแรก

	**LC-002: ระเบียบมหาวิทยาลัย (University IT Policy)**

* ระบบต้องรองรับการเชื่อมต่อกับระบบ Login กลาง (SSO/LDAP) ตามมาตรฐานความปลอดภัยของสำนักคอมพิวเตอร์  
* ห้ามเก็บรหัสผ่าน (Password) ของนิสิต/บุคลากรไว้ในฐานข้อมูลของระบบนี้เด็ดขาด (ใช้การ Verify ผ่าน API เท่านั้น)

## **7\. ดัชนี (INDEX)**

* **Admin Dashboard:** แผงควบคุมผู้ดูแลระบบ  
* **API (Application Programming Interface):** ช่องทางการเชื่อมต่อระหว่างซอฟต์แวร์

* **Backup:** การสำรองข้อมูล  
* **Business Rules:** กฎทางธุรกิจของระบบ  
* **Camera Integration:** การเชื่อมต่อกล้องถ่ายภาพ  
* **Database (PostgreSQL):** ฐานข้อมูล  
* **Email Notification:** การแจ้งเตือนผ่านอีเมล

* **Functional Requirements:** ความต้องการเชิงหน้าที่  
* **Hardware Interfaces:** อินเทอร์เฟซฮาร์ดแวร์  
* **HTTPS (TLS 1.2/1.3):** โปรโตคอลความปลอดภัย  
* **Image Processing:** การประมวลผลรูปภาพ  
* **JSON (JavaScript Object Notation):** รูปแบบข้อมูล  
* **LDAP / Active Directory:** ระบบฐานข้อมูลผู้ใช้ของมหาวิทยาลัย  
* **Master Data:** ข้อมูลหลัก (เช่น รายชื่ออาคาร, คณะ)  
* **Non-Functional Requirements:** ความต้องการที่ไม่ใช่เชิงหน้าที่  
* **Notification System:** ระบบแจ้งเตือน

	**ตัวอักษร O-Z**

* **Officer (Staff):** เจ้าหน้าที่มหาวิทยาลัย/ผู้รับผิดชอบงาน  
* **PDPA:** พระราชบัญญัติคุ้มครองข้อมูลส่วนบุคคล  
* **RBAC (Role-Based Access Control):** การจัดการสิทธิ์ตามบทบาท  
* **React:** ไลบรารีสำหรับสร้าง User Interface  
* **Reporting:** การออกรายงาน  
* **Samo (Student Union):** สโมสรนิสิตคณะ (ผู้คัดกรองงาน)  
* **Security:** ความปลอดภัยของระบบ  
* **Ticket ID:** เลขที่ใบคำร้อง  
* **User Interface (UI):** ส่วนติดต่อผู้ใช้  
* **User Story:** เรื่องราวผู้ใช้งาน  
* **Workflow:** กระบวนการไหลของงาน (Pending \-\> In Progress \-\> Completed)
